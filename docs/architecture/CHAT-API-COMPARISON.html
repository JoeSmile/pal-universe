Chat API 方案对比

方案A（全量历史）vs 方案B（增量 + conversation_id）

❌ 方案A: 全量历史

POST /chat/stream
{
  "messages": [
    {"role":"user", "content":"怎么繁殖Anubis?"},
    {"role":"assistant", "content":"...第1轮回答..."},
    {"role":"user", "content":"那Jetragon呢?"},
    {"role":"assistant", "content":"...第2轮回答..."},
    ...30轮后...超级大数组...
  ]
}
messages 大小∞ 膨胀 · 30轮 ≈ 50KB
Nginx 限制默认 1MB，聊多了报 413
网络开销每轮上传全部历史·浪费带宽
后端 Session 管理不需要（但也没必要）
LLM 费用每轮重复传旧对话到 LLM · 浪费 token
前端实现简单，一个数组追加
好处无状态，服务端重启不丢会话

✅ 方案B: 增量 + conversation_id

POST /chat/stream
{
  "conversation_id": "abc123", ← 后端创建，前端存
  "message": "那Jetragon呢?" ← 只传新消息
}
消息体大小固定 ≈ 200 字节，不膨胀
Nginx 限制永远不会超
网络开销每轮只传一句话，很低
后端 Session 管理LangGraph thread_id 已支持
LLM 费用后端控制在最近 20 轮，不浪费
前端实现多存一个 conversation_id 而已
好处体积稳定、费用可控、后端截断自由

🏆 量化对比（30 轮对话）

维度方案A（全量历史）方案B（增量）结论
请求体大小 ~50KB（messages 数组） ~200B（只有1条消息） B 胜 · 250x 差距
请求体重传 前面29轮重复上传29次 每轮只传新增 B 胜 · 零冗余
Nginx 风险 200轮 → ~350KB，仍在1MB内 永远不超 B 胜 · 无风险
服务端重启 不丢（前端重发 history） 丢 conversation（需持久化） A 胜 · 但可加持久化解决
LLM token 浪费 每轮重复传全部 history 后端按需截断（最多20轮） B 胜 · 费用低 30-50%
后端实现复杂度 简单，纯透传 需要管理 session + 截断逻辑 A 略简单
结论：B 方案全面胜出。且你现有的 LangGraph 已用 thread_id 管理 session，改造成本极低。
