# Pal Universe — AI Agent 架构设计

> 覆盖：RAG 兜底、缓存策略、工具链主动搜索、多步推理
> 后端技术栈：LangGraph StateGraph + FastAPI + pgvector + DeepSeek

## 二、RAG 兜底策略（拿不到答案怎么办）

### 2.1 三层相似度阈值

```python
SIMILARITY_THRESHOLDS = {
    "exact": 0.8,   # 精确匹配 → 直接返回
    "fuzzy": 0.6,   # 模糊匹配 → 返回 + 标注"可能相关"
    "broad": 0.4,   # 宽泛匹配 → 返回 + 建议用户补充信息
}
```

### 2.2 降级流水线

```
pgvector 检索 score < 0.4 (无结果)
  │
  ├─ Step 1: 同义词扩展后重试
  │   "Anubis" → "冥王犬"、"葬祭狗"
  │   帕鲁名 → 中英文互查
  │
  ├─ Step 2: 降级为关键词全文搜索 (pg_trgm)
  │   pgvector 语义没找到 → 试试 PostgreSQL FTS
  │
  ├─ Step 3: 触发工具链 (见下节)
  │   Agent 主动调用 web_search 查攻略
  │
  └─ Step 4: 诚实回答
      "抱歉，我还没学到这个信息。建议你试试搜索「关键词」"
      + 推荐 3 个相关帕鲁名让用户继续问
```

### 2.3 诚实回退原则

LLM **绝不编造数据**。当所有检索途径都失败时：

```
你: "Palworld 1.0 新增了哪些帕鲁？"

Agent 内部:
  1. pgvector 搜索 "1.0 新增 帕鲁" → 无结果
  2. 同义词扩展 "new pals v1.0" → 无结果  
  3. 调用 web_search("Palworld 1.0 new pals list")
  4. 从搜索结果提取 → 组织回答 → 标注来源

回答:
"根据网上搜索到的信息，Palworld 1.0 新增了 47 只帕鲁，
包括 Pupperai、Clovee、Amione 等。
(数据来源：网络搜索，建议以游戏内实际内容为准)"
```

## 三、缓存策略

### 3.1 两级缓存架构

```
                   ┌──────────────┐
                   │  用户问题     │
                   └──────┬───────┘
                          │
              ┌───────────┼───────────┐
              │           │           │
         ┌────▼────┐ ┌───▼───┐ ┌────▼────┐
         │ Valkey  │ │ 本地  │ │ 浏览器  │
         │ 服务端   │ │ LRU   │ │ Cache   │
         │ 缓存    │ │ 内存  │ │ API     │
         └─────────┘ └───────┘ └─────────┘
              │           │           │
              └───────────┼───────────┘
                          │
                    ┌─────▼─────┐
                    │  缓存命中?  │
                    │  → 直接返回 │
                    └───────────┘
```

| 层级 | 存储 | 容量 | TTL | 命中率预估 |
|------|------|------|-----|-----------|
| L1: 浏览器 | localStorage | 50 条 | 24h | ~15% |
| L2: 本地 LRU | Python dict | 200 条 | 1h | ~25% |
| L3: Valkey | Redis 兼容 | 1000 条 | 24h | ~30% |

### 3.2 缓存 Key 设计

```python
def cache_key(question: str) -> str:
    """归一化 + hash"""
    q = question.lower().strip()
    q = re.sub(r'[^\w\u4e00-\u9fff]', '', q)  # 去标点，保留中文
    return hashlib.md5(q.encode()).hexdigest()
```

### 3.3 缓存预热

上线后自动缓存热门前 100 问题：

```python
HOT_QUESTIONS = [
    "Anubis 怎么繁殖",
    "哪些帕鲁适合挖矿",
    "前期抓什么帕鲁好",
    "Jetragon 在哪抓",
    "怎么配火属性队伍",
    # ... 从日志中持续补充
]
```

启动时自动查询 → 缓存结果 → 用户来直接命中。

## 四、工具链主动搜索

这是 LangGraph Agent 最强大的能力——**Agent 可以自主决定调用什么工具、按什么顺序调用**。

### 4.1 可用工具清单

| 工具 | 功能 | 数据源 | 触发场景 |
|------|------|--------|---------|
| `pal_search` | 查询帕鲁属性/技能/掉落 | pgvector / JSON | "Anubis 什么属性" |
| `breeding_calc` | 繁殖计算 | pgvector + 公式 | "怎么繁殖 Jetragon" |
| `breeding_reverse` | 反向查父代 | pgvector | "哪些组合能生出 Anubis" |
| `map_search` | 帕鲁位置 | atlas-data JSON | "炎龙在哪抓" |
| `web_search` | 网络搜索（已有） | DuckDuckGo | 数据库查不到的信息 |
| `type_chart` | 属性克制表 | JSON | "什么属性克暗系" |

### 4.2 多步推理示例

这是跟普通 Chat Bot 最大的区别——**Agent 可以拆解问题、多步执行**：

```
用户: "我想组一个火属性队伍打暗系 Boss，推荐什么帕鲁？"

Agent 推理:
  Step 1: 查 type_chart("dark") → 暗系被龙系克制
  Step 2: 查 pal_search(elements="dragon") → 列出所有龙系帕鲁
  Step 3: 按攻击力排序 → 取 Top 5
  Step 4: 查每个帕鲁的 skills → 筛选有暗系技能的
  Step 5: 组织回答

回答:
"暗系 Boss 被龙系克制（×2 伤害）。推荐队伍：
1. Jetragon (龙+暗) - ATK 140, 技能: Dark Laser
2. Astegon (龙) - ATK 150, 技能: Dragon Breath
3. Shadowbeak (暗) - ATK 130, 技能: Nightmare Ball
..."
```

### 4.3 Agent 决策流程

```
LLM 收到用户问题
  │
  ├─ 需要查数据？ → 调用 pal_search / breeding_calc
  │     └─ 结果不够？ → 再调另一个工具（链式调用）
  │
  ├─ 需要实时信息？ → 调用 web_search
  │     └─ 提取有用信息 → 组织回答
  │
  └─ 直接能回答？ → 直接返回（常见问题已缓存）
```

LangGraph 的 `Command(goto="tool_call")` 机制天然支持这种多步推理——Agent 可以反复进入 `chat → tool_call → chat → tool_call → ...` 直到信息足够。

### 4.4 工具链成本控制

| 情况 | 动作 | 成本 |
|------|------|------|
| 本地 JSON 能回答 | 不调 LLM | $0 |
| 需要调一个工具 | 只调一次 LLM 判断 | ~$0.0001 |
| 需要链式 3 个工具 | 调 4 次 LLM（1次判断+3次总结） | ~$0.0005 |
| 需要 web_search | 新增 1 次 LLM 总结 | ~$0.0002 |

**绝大多数问题在 1-2 步工具调用内解决。** 超过 4 步的复杂问题才可能到 $0.001。

## 五、System Prompt 设计

```markdown
你是 Pal Universe 的 AI 助手，精通 Palworld 全部 299 只帕鲁的
数据、繁殖系统和游戏机制。

## 回答规则

1. 有数据支撑 → 准确回答，标注数据版本
2. 数据不够 → 调用工具查 (pal_search / breeding_calc / web_search)
3. 工具也查不到 → 诚实说不知道，推荐搜索关键词
4. 绝不编造数据、绝不虚构帕鲁名称或属性

## 工具使用原则

- 查帕鲁属性 → pal_search
- 查繁殖 → breeding_calc 或 breeding_reverse
- 需要实时攻略 → web_search
- 复杂问题 → 拆成多步，链式调用工具

## 输出格式

- 提及帕鲁时用中文名·英文名（如 冥王犬·Anubis）
- 关键信息加粗
- 提供后续快捷操作建议
```
